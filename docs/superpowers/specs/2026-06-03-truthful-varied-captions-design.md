# Truthful, Varied Caption & Hook Generation

**Date:** 2026-06-03
**Status:** Approved (design)
**Scope:** `scripts/generate-script.ts` + new `scripts/lib/caption-framing.ts`

## Problem

The TikTok post caption and opening hook overlay are generated by a single Claude
call in `scripts/generate-script.ts`. The system prompt frames every reveal as
"vivid family memories" (line 53) and tells Claude to "reference specific decades,
family relationships" (line 62) in a "warm, personal" first-person voice.

But the reveal photos are **AI-invented demonstration scenarios** (self-sourced via
`scripts/generate-reveal-photos.ts`), not the poster's real family. The result is
captions that falsely claim personal ownership — e.g. "My grandmother's wedding
photo, restored after 60 years." This is untrue and the user wants it fixed.

A secondary goal: the feed currently feels same-y because every caption uses the
same emotional first-person register. We want genuine variety across videos.

## Goals

1. **Truthful** — captions/hooks never claim the poster personally owns or found the
   photo. No fabricated "my grandmother" / "I found this in my attic" anecdotes.
2. **Varied** — reveal captions rotate across distinct truthful framings so the feed
   doesn't feel repetitive.
3. **Focused** — change only the post caption + hook generation. No schema, DB, API,
   or dashboard changes.

## Non-Goals / Out of Scope

- Per-pair on-screen overlay captions (`caption_before` / `caption_after` in
  `scripts/lib/caption-text.ts`) — left untouched.
- Tip-content voice — tips keep their current educational/expert register and are NOT
  rotated. (They are still bound by the global truthfulness rule, but tips don't make
  family-ownership claims today.)
- Any dashboard UI for selecting/overriding framing (considered as approach "C",
  deferred; can be added later if desired).

## Why this approach (B: code-selected framing)

Each script generation is a separate, **stateless** Claude call — it cannot see the
other videos in the feed. Telling Claude to "vary the framing" inside one prompt is
unreliable: stateless calls converge on whatever feels most natural (emotional
third-person), so the feed stays same-y. To get real variety, the *selection* must
happen in code and be injected into the prompt. Selecting in code also makes the
picker a pure function we can unit-test.

## Design

### New module: `scripts/lib/caption-framing.ts` (pure, testable)

```ts
export type CaptionFraming = 'capability' | 'third_person' | 'invitation';
```

Definitions table (weight = selection probability):

| Framing        | Weight | Voice                                   | Example hook                          |
|----------------|--------|-----------------------------------------|---------------------------------------|
| `third_person` | 40%    | Honest emotional story, no "my/I"       | "Water-damaged for 60 years. Not anymore." |
| `capability`   | 35%    | Product / AI-demo forward               | "Old photo → restored by AI in seconds" |
| `invitation`   | 25%    | Speaks to the viewer's own photos       | "Got photos like this in a drawer?"   |

Exports:

- `pickFraming(rng: () => number = Math.random): CaptionFraming`
  Weighted random selection. `rng` is injectable so tests can pin the result by
  feeding fixed values that land in each weight bucket.
- `framingInstruction(f: CaptionFraming): string`
  Returns the prompt snippet injected into the reveal user prompt, describing the
  voice + an example for that framing.

Weights live as a single source of truth (e.g. an ordered array of
`{ framing, weight, instruction }`) consumed by both `pickFraming` and
`framingInstruction`.

### `scripts/generate-script.ts` changes

1. **System prompt** (lines 51–76):
   - Remove the "vivid family memories" wording (line 53) and the "family
     relationships" instruction (line 62) that induce false ownership. Replace with
     neutral language about old/damaged photos being transformed.
   - Add a hard truthfulness rule:
     > These are demonstration photos showcasing the app — NOT the poster's own
     > family. NEVER write in first person claiming personal ownership (no "my
     > grandmother", no "I found this in my attic"). Never fabricate that the poster
     > personally found or owns the photo. Tell the story honestly in the assigned
     > voice.

2. **`buildUserPrompt(item, framing?)`**:
   - For `reveal`, inject `framingInstruction(framing)` into the prompt so Claude
     writes in the selected voice.
   - For `tip`, unchanged (no framing parameter used).

3. **`generateScript(item)`**:
   - For `reveal`, call `pickFraming()` and pass the result to `buildUserPrompt`.
   - `console.log` the chosen framing so the rotation is observable in pipeline logs.
   - Return shape (`GeneratedScript`) is unchanged.

### Data flow (unchanged except prompt text)

`generateScript(item)` → (reveal) `pickFraming()` → `buildUserPrompt(item, framing)`
→ Claude call → parse JSON → `GeneratedScript`. Persisted fields
(`hook_text`, `caption`, `hashtags`, ...) and all downstream posting logic are
untouched.

### Error handling

No new failure modes. `pickFraming`/`framingInstruction` are total functions over a
closed enum. If `framing` is somehow undefined for a reveal, `buildUserPrompt` falls
back to no framing snippet (current behavior) rather than throwing.

## Testing

New `npm test` cases (pure functions, no network):

- `pickFraming` returns `third_person` / `capability` / `invitation` for rng values
  landing in each weight bucket (boundaries of 0.40 and 0.75), and never returns an
  out-of-enum value.
- `framingInstruction` returns distinct, non-empty strings for all three framings.
- `buildUserPrompt` for a reveal item contains the injected framing instruction;
  for a tip item it does not.

## Risks

- Low. The only behavioral change is prompt wording and which framing snippet is
  selected. Worst case is caption tone differs from before — which is the intent.
